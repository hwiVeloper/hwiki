# LLM Wiki

- **유형**: concept
- **카테고리**: 개발

## 정의

LLM Wiki는 Andrej Karpathy가 제안한 LLM 기반 개인 지식 베이스 패턴이다. 기존 RAG(Retrieval-Augmented Generation) 방식이 매번 원본 문서를 검색하는 것과 달리, AI가 원본 자료를 읽고 영구적인 마크다운 위키로 **사전 컴파일(pre-compile)** 하는 접근법이다.

핵심 철학은 "Retrieve 하지 말고, Compile 하라"로 요약된다.

## 핵심 구조: 3-Tier 아키텍처

1. **Raw Sources** — 웹 클리핑, PDF, 메모 등 날것의 원본 자료. 불변(immutable) 데이터로 AI는 읽기만 한다.
2. **Wiki** — AI가 100% 소유하고 유지보수하는 마크다운 파일 디렉토리. 엔티티, 개념, 요약이 `[[wikilink]]`로 풍부히 연결된 상태.
3. **Schema** — AI의 동작 규칙, 폴더 구조, 링크 컨벤션 등을 정의하는 시스템 프롬프트 파일 (예: CLAUDE.md).

## 3대 Operation

- **Ingest**: 원본 자료를 분석하여 source 페이지를 생성하고, 관련 entity/concept 페이지를 동시 업데이트. 지식 컴파일 과정.
- **Query**: 목차(index)를 우선 스캔하여 필요한 위키 페이지만 참조. 탐색 결과가 새 위키 페이지로 영구 저장되어 지식이 축적됨.
- **Lint**: 모순 탐지, 고아 문서 복구, 빈 공간 식별, 오래된 정보 갱신 등 위키의 건강 상태를 점검하고 유지보수.

## RAG와의 차이

| 관점 | RAG | LLM Wiki |
|---|---|---|
| 데이터 처리 | 질문 시점에 원문 검색 | 사전에 위키로 컴파일 |
| 토큰 효율성 | 매번 원문 전체 참조 | 정제된 요약본만 참조 |
| 지식 연결 | 단방향 (문서→답변) | 양방향 (자동 백링크, 교차 참조) |
| 축적 곡선 | 수평 (매번 휘발) | J커브 (복리 축적) |

## 지식의 복리 효과 (Compounding)

정보가 추가될 때마다 기존 위키와 연결되며 가치가 상승한다. 유지보수 비용(Bookkeeping Cost)이 0에 수렴하므로, 기존에 포기했던 개인 위키/노트 정리가 지속 가능해진다.

## 구현 도구

- **프론트엔드**: Obsidian — 시각화된 그래프 뷰로 탐색하고 연결된 링크를 브라우징
- **백엔드**: Claude Code — 북키핑, 요약, 태깅, 연결, 모순 검출을 백그라운드에서 수행

## 스케일링

위키 규모가 100개 이상으로 커지면 index.md 스캔만으로는 컨텍스트 윈도우 한계에 부딪힌다. 이 때 Vector Embeddings + BM25 + Cross-encoder Reranking을 MCP 서버로 연결하여 로컬에서 50ms 이내 검색을 실현할 수 있다.
